# 东软环保公众监督系统前端架构设计文档

## 一、设计目标

本项目前端采用四端基础版设计，包含公众监督员端、网格员端、系统管理员端、决策者端。

前端架构目标如下：

1. 页面结构清晰，能够直观看出四端职责。
2. 公共能力统一复用，避免每个端重复写相似代码。
3. 接口层按后端业务资源拆分，便于后端同学对照开发。
4. 后端接口未完成时，可以先使用 mock 数据支撑页面展示。
5. 后续页面、路由、接口、图表能够继续扩展。

## 二、技术选型

| 技术 | 用途 |
| --- | --- |
| Vue3 | 前端页面开发和响应式数据管理 |
| Vite | 前端构建工具和开发服务器 |
| Element Plus | 表单、按钮、表格、弹窗、布局等 UI 组件 |
| Vue Router | 四端页面路由管理 |
| Axios | 前后端接口请求 |
| ECharts | 决策者端数据可视化图表 |

当前已安装依赖：

```bash
npm install element-plus @element-plus/icons-vue
npm install vue-router axios echarts
```

## 三、目录结构设计

项目采用“页面按端划分，接口按资源划分，公共内容集中复用”的结构。

```txt
src/
  api/
  assets/
  components/
  layouts/
  mock/
  router/
  utils/
  views/
    supervisor/
    grid/
    admin/
    decision/
```

### 1. 页面目录 `views`

`views` 目录只放页面级组件，按照四端角色拆分：

| 目录 | 负责端 | 说明 |
| --- | --- | --- |
| `views/supervisor` | 公众监督员端 | 注册、登录、提交反馈、历史反馈 |
| `views/grid` | 网格员端 | 登录、任务列表、任务详情、检测确认 |
| `views/admin` | 系统管理员端 | 登录、反馈管理、任务指派、统计管理 |
| `views/decision` | 决策者端 | 数据可视化大屏 |

这样划分的优点是四端边界清楚，便于老师检查和团队协作。

### 2. 接口目录 `api`

`api` 目录按后端业务资源拆分，不按前端角色拆分。

计划文件如下：

| 文件 | 说明 |
| --- | --- |
| `request.js` | Axios 实例、请求拦截、响应拦截、错误处理 |
| `supervisor.js` | 公众监督员注册、登录、信息查询 |
| `aqiFeedback.js` | 反馈提交、反馈查询、反馈指派 |
| `gridMember.js` | 网格员登录、网格员查询 |
| `statistics.js` | 检测数据提交、统计数据查询 |
| `admins.js` | 管理员登录 |

这样划分的原因是：同一个接口资源可能被多个端使用。例如 `aqiFeedback` 既会被公众监督员端用于提交反馈，也会被管理员端用于管理反馈。

### 3. 公共组件目录 `components`

公共组件用于复用多个页面中的相似功能，避免重复开发。

后续建议组件：

| 组件 | 用途 |
| --- | --- |
| `AqiLevelTag.vue` | 展示 AQI 等级标签和颜色 |
| `FeedbackForm.vue` | 空气质量反馈表单 |
| `FeedbackTable.vue` | 反馈数据表格 |
| `RoleCard.vue` | 首页角色入口卡片 |
| `EmptyState.vue` | 空数据展示 |
| `PageHeader.vue` | 页面标题和操作按钮区域 |

复用原则：

1. 两个以上页面出现相似结构时，再抽成公共组件。
2. 页面特有的业务逻辑保留在页面中。
3. 公共组件通过 props 控制显示内容，不直接依赖某一个端。

### 4. 布局目录 `layouts`

`layouts` 目录放页面外壳，不放具体业务逻辑。

计划布局如下：

| 布局 | 用途 |
| --- | --- |
| `PublicLayout.vue` | 公众监督员端，移动端优先 |
| `ConsoleLayout.vue` | 网格员端、系统管理员端，偏后台管理风格 |
| `DashboardLayout.vue` | 决策者数据大屏 |

### 5. 工具目录 `utils`

`utils` 目录存放纯逻辑工具函数。

计划文件如下：

| 文件 | 用途 |
| --- | --- |
| `auth.js` | 登录信息存取、角色判断、退出登录 |
| `aqi.js` | AQI 等级计算、颜色映射 |
| `format.js` | 日期、状态、文本格式化 |

### 6. Mock 数据目录 `mock`

`mock` 目录用于后端接口未完成时提供模拟数据。

计划文件如下：

| 文件 | 用途 |
| --- | --- |
| `feedback.js` | 反馈列表、反馈详情模拟数据 |
| `grid.js` | 省、市、网格员模拟数据 |
| `statistics.js` | 图表统计模拟数据 |

Mock 数据只在开发阶段使用。后端接口完成后，页面不直接改动，只在 `api` 层切换为真实请求。

## 五、路由设计

路由按四端前缀划分：

| 端 | 路由前缀 |
| --- | --- |
| 公众监督员端 | `/supervisor` |
| 网格员端 | `/grid` |
| 系统管理员端 | `/admin` |
| 决策者端 | `/decision` |

后续路由守卫规则：

1. 未登录用户只能访问登录、注册和入口页。
2. 已登录用户根据 `role` 访问对应端页面。
3. 访问无权限页面时跳转到对应登录页或首页。

角色标识建议：

| 角色 | role 值 |
| --- | --- |
| 公众监督员 | `supervisor` |
| 网格员 | `grid` |
| 系统管理员 | `admin` |
| 决策者 | `decision` |

## 六、接口设计原则

前端统一通过 `src/api/request.js` 调用后端接口。

统一约定：

| 项 | 约定 |
| --- | --- |
| 后端接口前缀 | `/api` |
| 查询类请求 | GET |
| 提交类请求 | POST |
| 请求数据格式 | JSON |
| 返回数据格式 | `{ code, message, data }` |
| 成功状态 | `code === 200` |

页面不直接写 `axios.get` 或 `axios.post`。页面只调用接口函数，例如：

```js
saveAqiFeedback(payload)
listAqiFeedbackBySupervisorId(supervisorId)
```

这样后端地址或返回格式变化时，只需要修改接口层。

## 七、数据流设计

基础数据流如下：

1. 用户在页面填写表单。
2. 页面进行基础表单校验。
3. 页面调用 `api` 中的接口函数。
4. `api` 函数通过 `request.js` 发送请求。
5. 后端返回 `{ code, message, data }`。
6. 页面根据结果展示成功提示、错误提示或列表数据。

在后端未完成前：

1. 页面仍调用 `api` 函数。
2. `api` 函数临时返回 mock 数据。
3. 后端完成后替换 `api` 文件内部实现。
4. 页面组件不需要大改。

## 八、开发顺序建议

建议按业务闭环开发：

1. 补齐路由和 Axios 基础层。
2. 完成公众监督员端：注册、登录、提交反馈、历史反馈。
3. 完成管理员端：查看反馈、指派网格员。
4. 完成网格员端：查看任务、提交检测确认。
5. 完成决策者端：统计图表展示。
6. 统一做移动端适配、表单校验和接口联调。

这样的顺序可以尽早跑通核心业务流程：公众提交反馈，管理员指派任务，网格员确认数据，决策者查看统计。
